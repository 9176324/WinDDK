Co-installer for Copying INF Files

[This is preliminary documentation and subject to change.]

Abstract: This document accompanies the redistributable co-installer that can be used to build an INF file that can be used to copy multiple INF files. This binary is to support the CopyINF directive on the Microsoft® Windows® 2000 operating system. Support for the CopyINF directive discussed in this document is built into the Windows XP operating system, so this co-installer need not be used in an INF section targeted for only Windows XP. The co-installer must not be used when installing on a 64-bit platform.

SUMMARY

This document accompanies and describes a redistributable co-installer that can be used to install additional device INF files on the target system during a device installation. The instructions provided herein apply to the Microsoft Windows 2000 and Windows XP operating systems.

The co-installer described in this article interprets CopyINF directives in a [DDInstall] section in an INF file. Using CopyINF together with this co-installer allows for one INF file to copy multiple INF files to the target system.

This technique is useful, for example, when installing a multifunction device. Using this method, the base INF file for a multifunction device could list the INF files for the individual functions of the devices in a CopyINF directive and install the co-installer, which would then install the referenced INF files for the individual functions.

The syntax of the CopyINF directive is:

[DDInstall]
CopyINF = filename.inf[,filename2.inf]...

The source code for this co-installer and two sample INF files are available at <ddkroot>\src\setup\cocpyinf and <ddkroot>\src\setup\cocpyinf\install respectively.

Notes:

  • The INF copying functionality demonstrated in this example is not necessary for Windows XP, which includes built-in support for the CopyINF directive. However it is recommended for an INF that may be installed on Windows® 2000. For an installation that may include a 64-bit platform, decorated sections should be used to ensure that this co-installer is not installed and used on a 64-bit platform as native platform co-installers must be used.

  • Windows 2000 and Windows XP Setup expect the locations of INF files specified in the CopyINF directive to be relative to the source location of the INF in which they are specified.

  • The INF file copy occurs after the device has been installed. Windows 2000 and Windows XP Setup assume that the source media is still available. Therefore, be sure to include all INF files on each disk of a multi-disk installation package.

  • If the current INF file name and the original INF file name are different, then CoCpyInf assumes the INF has already been copied and will not perform the file copies.

When a device installation from a third-party INF occurs, that INF is copied to the system INF directory under a unique OEM<n>.inf name. The co-installer determines whether the additional CopyINF actions are necessary, depending on whether the INF’s original name differs from its current name. If the two names are the same, then the INF has not yet been installed into the INF directory, so the additional INF files are copied. If the INF’s original and current names differ, this indicates installation is from a third-party INF that already resides in the INF directory. In that case, the additional INF copy operations do not need to be performed, because they would have been copied previously.

Attempting to perform the additional copies in this scenario wouldn’t work anyway, because the third-party media is no longer available.

To ensure that the names of the INF files are always different to avoid support issues, do not use names like OEM<n>.inf (e.g. OEM1.INF), or a name that is the same as the name of a system provided INF file.

  • Some multifunction devices do not have a single base INF file. For example, for a multifunction PCI card, all functions are enumerated as peers of each other. Therefore, a co-installer created for such a device must be registered for each function, and each function’s installation would have to reference the other INFs, because there is no guarantee as to which function will be enumerated first.

USING THE CO-INSTALLER

To use the co-installer for driver installation and registration:

Copy the 32-bit Cocpyinf.dll redistributable binary to the directory where you are assembling your driver package. You should use this following version of cocpyinf.dll:

tools\coinstallers\x86\cocpyinf.dll

To use this co-installer, add the following sections to every INF file for your drivers. If the multifunction device is a single parent that enumerates multiple children, then only the parent’s INF needs to install or use the co-installer.

·         Add a [DDInstall.CoInstallers] section for each device specified in the INF. This section must include the following two lines:

AddReg = CoCopyINF.AddReg
CopyFiles= CoCopyINF.CopyFiles

For example:

[InstallA.NT.CoInstallers]
AddReg = CoCopyINF.AddReg
CopyFiles= CoCopyINF.CopyFiles

 ·         Add the following sections:

[CoCopyINF.AddReg]
HKR,,CoInstallers32,0x10000,"cocpyinf.DLL,CoCopyINF"

[CoCopyINF.CopyFiles]
cocpyinf.dll,,,0x10

·         Add a reference to the CopyFiles section in the [DestinationDirs] section. For example:

[DestinationDirs]
CoCopyINF.CopyFiles = 11

·         Add references to all the other INF files in the package in the [DDInstall] section. For example:

[InstallA.NT]
CopyInf = B.INF

·         Add sections to install the drivers.

For more information, see the sample INF files located at <ddkroot>\src\setup\cocpyinf\install.

TESTING

To ensure that the INF files are not in the system, type:

Del %windir%\INF\OEM*.?NF

Note that this step is only safe to do for testing purposes, and will delete any other third party INF files that are present on the system. The system may be left in an unstable state.

Uninstall the devices in Device Manager.

Install the devices by clicking on the "Scan for hardware changes" item in the Device Manager Action Menu.

Once all the relevant devices have been installed, you should check that a re-install also works. Uninstall the devices in Device Manager again (this time without removing the OEM*.?NF files) and click on "Scan for hardware changes" again.
