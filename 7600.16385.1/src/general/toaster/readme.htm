Toaster – Driver Learning Package

 

Toaster directory contains a huge collection of samples meant to demonstrate most of the fundamental aspects of Windows driver development. All the samples are built for a hypothetical computer bus, called “Toaster Bus”, meant for connecting toasters devices to PCs.

Understanding the structure and working of all these various samples will provide a basic understanding of windows driver model and get you started on driver development.  At a high level, these samples are meant to teach you how to write:

·       Bus driver – one that controls the toaster bus and enumerates devices.

·       Function driver – one that talks to a toaster device connected to toaster bus.

·       Filter driver – how to write class & device filters that layer above and below the function driver to enhance or alter the I/O packets.

·       Coinstaller – Used to customize the installation of a specific instance of a device. It can also used to configure device properties through device-manager property page.

·       Classinstaller – Used to customize or provide additional properties to all the devices that plug to toaster bus.

·       Device metadata – Used to display device-specific information to the user in ‘Devices and Printers’ folder on Windows7 and later OSes.

·       How to interact from one driver to another driver (disjoint stack) in PNP friendly way.

·       How to enumerate  PnP devices in usermode application, and how to interact with them in pnp friendly way.

·       How to write INF files to install all the various drivers described above.

·       How to install custom application along with driver installation.

Some of the sample drivers are written using different technologies (WDM, KMDF, UMDF) and placed side-by-side to show the difference in the programming model. The table below describes what’s contained in each of the top-level directories.

WDM

This folder contains sample drivers that implement the Windows Driver Model. Drivers written in WDM are typically complex and less robust because they interface with OS directly and end up managing all the synchronization and concurrency issues around I/O and PnP/Power request.  Instead of writing drivers in WDM, you should write drivers in either KMDF or UMDF. The KMDF & UMDF folder contain samples that are functionally equivalent to the WDM drivers and are lot simpler to write and manage. Please read the toaster.htm file to learn more about the drivers under this folder.

KMDF

This folder contains sample drivers written using Kernel mode driver framework.  The drivers in this folder are functionally equivalent to WDM drivers.  Please read the toaster.htm file to learn more about the drivers under this folder.

UMDF

This folder contains sample drivers written using User mode driver framework.  The drivers in this folder (WUDFToaster & toastmon) are functionally equivalent to WDM drivers (featured and toastmon). There is no bus driver sample in UMDF because UMDF doesn’t support writing bus drivers.  Please read the Windows Driver Foundation book to learn the difference between KMDF and UMDF and how to decide which framework to use. Please read the toaster.htm file to learn more about the drivers under this folder.

ClassInstaller

This folder contains a sample code to show how to write a class installer DLL. This DLL provides a custom icon for “Toaster” class and provides a customer property sheet in the device manager to change the friendly name of the device. This DLL is referred in the INF file used to install driver for toaster device.

CoInstaller

This folder contains a sample code to show how to write a coinstaller DLL. This DLL shows how to create a friendly name based on the instance number of the device and also how parse custom section in the INF. This DLL is referred in the INF file used to install driver for toaster device.

Devicemetadatapackage

The sample files under this folder show how to create a device metadata package for Toaster devices.

Device metadata contains the information that describes the device, including the following:

•            The vendor name.

•            The model name and description of the device.

•            One or more device categories.

A device metadata package consists of multiple XML documents, with each document specifying various components of the device’s attributes. Starting with Windows® 7, the Devices and Printers folder uses these XML documents to display device-specific information to the user. Through these XML documents, the vendor can customize how this information appears, as well as the specific information that is included.

 

This sample has the following:

·         The source XML metadata files and device icon file for the English and Japanese locales.

·         The compiled device metadata package for both the English and Japanese locales.

 

For more information on compiling metadata packages, refer to the “Building Device Metadata Packages” section in the WDK.

 

To install the device metadata package on Windows 7:

 

  1. Copy the device metadata package from the floppy drive or temporary file in the target system.
  2. If the target system is in the Japanese locale, paste it into %programdata%\microsoft\windows\devicemetadatastore\ja-jp.
  3. If the target system is not in the Japanese locale, paste it into %programdata%\microsoft\windows\devicemetadatastore\en-us.
  4. Note that this action requires administrator privilege. Elevate the privilege when you are prompted by entering the administrator’s user name and password.

After installing a toaster device, open the “Devices and Printers”. You will see the sample toaster icon and sample information for the toaster sample.

More information on how to install device metadata packages is introduced in the device metadata package section in WDK document.

Exe

Exes under this folder are used to interact with the toaster bus driver and function driver. Since the drivers, be it written in WDM, KMDF or UMDF, are functionally equivalent and expose same interfaces,  these application work with all of them.

 

This directory contains three subdirectories notify, enum, and toast:

  • Enum.exe is a user-mode enumerator, a simple console application. Because the toaster bus is not real, you use this application to tell the bus driver to plug in, unplug, and eject devices from the system.  Type Enum.exe to see the usage.
  • Toast.exe: This is a user-mode console application to control the toaster.  This application enumerates toaster device open the last enumerated device and sends read request to it.
  • Notify.exe: This GUI application not only combines the functionality of Enum.exe and toast.exe but also shows how to handle PnP notification in user-mode. You should install the coinstaller for the toaster device by using toastco.inf to get meaningful display of PnP notification. Another cool feature of this exe is that you can specify some other HW ID (Instead of the default toaster device id) and cause other drivers to be loaded as a function driver.

ToastPkg – Toaster Installation Package

This sample describes how you can install your driver package on an end user system in a consistent, robust manner that works cooperatively with Plug and Play. This has the sources for an integrated device installation solution that illustrates how to provide support for the various user approaches to adding new hardware, while providing optional value-added software. Please read toastpkg.htm for more information on this sample.
