Virtual Serial Driver

 

SUMMARY

The Vserial sample driver is a test tool for serial digitizer drivers. The driver replaces Serial.sys in the driver stack for a serial digitizer. Vserial  reads data from a file (instead of the serial port) so  you can test a serial digitizer driver like Wacompen.sys (a WDK sample available in the \src\input\hiddigi directory) on a computer that doesn’t have the digitizer. The driver can fail many of the calls that the digitizer driver makes to the serial driver. This ability helps you test the error handling code in the digitizer driver.

BUILDING THE SAMPLE

The sample driver can be built using the following build environments:

  • Windows Vista build environment (checked or free)
  • Windows Server 2003 build environment (checked or free)
  • Windows XP build environment (checked or free)
  • Windows 2000 build environment (checked or free)

To build the sample, do the following:

  1. At the build environment command prompt, change the folder to the vserial folder (src\input\vserial).
  2. Enter “build –ceZ” to build vserial.sys.

INSTALLING THE SAMPLE

You can install the driver on any computer that is running  Microsoft Windows 2000 or a later version of the Windows operating system. You must modify the INF file for the digitizer driver to include the virtual serial (Vserial.sys) driver instead of Serial.sys. After you make this change, you can install the driver on any computer with the help of the devcon utility that is included in the WDK. To install the driver, run “-devcon install  <INF name>  <hardware ID>”.

TESTING A SERIAL DIGITIZER DRIVER USING THE SAMPLE

You must put device specific data in the data file to test normal functionality of the driver. You can put any random data in the file to test the driver’s error handling. After the virtual serial driver is loaded, it looks for data in the \\??\\C:\\TempDir\\SimPenData.log path. You can modify this by specifying a “FileName” registry value under the \\Registry\\Machine\\Software\\Microsoft\\SimDevice key. (Both paths are set in the source code so you can change them)

If you write certain values under \\Registry\\Machine\\Software\\Microsoft\\SimDevice , the driver reacts to IRPs in different ways. The Vserial.h file contains definitions for the following registry values:

#define READ_RESPONSE       L"ReadResponse"

#define WRITE_RESPONSE      L"WriteResponse"

#define CONTROL_RESPONSE    L"ControlResponse"

#define PNP_QUERY_RESPONSE  L"PnpQueryResponse"

#define CLOSE_RESPONSE      L"CloseResponse"

#define CREATE_RESPONSE     L"CreateResponse"

#define CLEANUP_RESPONSE    L"CleanupResponse"

 

These values are all DWORD values. When READ_RESPONSE or WRITE_RESPONSE is set to 0, all IRPs of the type will fail. When these values are set to a value that is greater than 0 then the nth IRP of that type will fail. If you want to fail the sixth read request that is sent to the serial driver, set the ReadResponse value to 6. For control codes, if you set the value of CONTROL_RESPONSE to one of the  relevant IOCTL s that is sent to the serial driver, the IOCTL request will fail.  

If CLOSE_RESPONSE, CREATE_RESPONSE or CLEANUP_RESPONSE appear  under the SimDevive registry key, the respective IRPs will fail regardless of the number that is assigned to the value.  You can modify these registry values from user mode to execute different test scenarios.
