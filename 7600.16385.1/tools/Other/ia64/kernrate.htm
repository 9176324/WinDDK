Kernrate Usage Guide

 

Purpose

 

Kernrate is a sample profiling tool meant to help identify primarily where CPU time is being spent. Both Kernel and user mode processes can be profiled separately or simultaneously. With proper support, Kernrate can also be used to profile CPU events (sources) other than time, depending on the CPU type.

 

Supported OS Platforms

 

The version of Kernrate documented here will run under Windows 2000, Windows XP and Windows 2003 servers. Compatibility: Some newer Kernrate options may not be available or supported on Windows 2000 or Windows XP.

 

Supported Hardware

 

The version of Kernrate documented here supports Intel x86 processors (Pentium and above), AMD equivalent processors as well as Intel 64 bit and AMD 64 bit platforms. The “Time” source is supported on all platforms. The degree of support for sources other than “Time” varies, depending on the amount of HAL support for the counters available with each processor type.

 

Method of Operation

 

Kernrate opens the process selected to be monitored and loads the process modules. After starting the profile, each module accumulates “hits” based on the number of CPU event occurrences that happened in the module address space. For pre-selected modules (“zoomed” modules), the module address space is divided into “buckets”. The default bucket size is currently 16 bytes (the minimum is 4 bytes). After initializing the profile, the kernel adds hit counts to the appropriate buckets, based on the addresses where CPU events were spent in. The frequency of the profile can be controlled by setting the number of event occurrences per hit. Therefore a setting of 2,000 event occurrences per hit will generate 10 times the sample rate as compared to a setting of 20,000 event occurrences per hit. After the profiling has ended, Kernrate translates the bucket addresses into symbols and performs all the necessary statistic calculations.

 

When To Use Kernrate

 

  1. Use Kernrate for preliminary identification of CPU usage patterns and CPU hogs down to API level (and even down to code sections within API’s to a limited extent).
  2. Use Kernrate for identifying specific CPU issues with profile sources other than the default (Time).
  3. Use Kernrate to measure the effect of code changes and performance improvements on CPU usage.
  4. There is little point in using Kernrate in cases where the bottleneck is not CPU related (low CPU usage), although the system-wide and process-specific summaries as well as lock information provided by Kernrate could help in initial identification of the culprits.

 

Limitations and Overhead Considerations

 

  1. Kernrate is a sampling profiler. It may miss very short lived events. Increasing the sampling rate may help, but this will cost in terms of increased interrupt-rate (in some cases the machine may become unresponsive for a long period). In general, short lived events have little influence on the average CPU usage (but they may be of interest for other reasons). Kernrate may also miss periodic events that occur at frequencies close to the sampling rate (but out of the sample). Selection of sampling rates that are not common multiples of clock frequencies can help in that case.
  2. The module address space is divided into buckets of no less than 4 bytes each (default is 16 bytes). A bucket address range may be inhabited by more than one API. Learning to use the –d, -r or –v 2 options will help to identify who really occupies every bucket. This issue is also important in case of code optimizations that restructure the module.
  3. The more processes being monitored, the more memory overhead. Besides the need to allocate memory for structures/arrays based on the number of processes, modules and sources, Kernrate also calls Dbghelp to load basic symbol information for the modules in the import list of every process being monitored at the beginning of the run. After profiling is done, Kernrate loads the appropriate deferred symbols for every zoomed module, but it releases each symbol file as soon as it is done with processing the symbols for any particular zoom module. The peak memory usage occurs during the processing phase, after profiling is done. Kernrate will also allocate more memory depending on the number of processors in the system and the bucket size. In some cases it may be useful to run the kernel profile separately from the user mode processes and divide the sampling of multiple user processes into several separate runs.
  4. Kernrate may miss some of very short lived processes or some of these may go away before profiling/processing is done. Kernrate is not able to profile new processes created after its initialization.
  5. Some of the optional data impose more overhead or slight delay, such as the summary of %CPU usage for all running processes on the system at the end of the profile (the –t command line option) or collecting lock contention information (the –x family of options and in particular the system lock information). Most of that overhead is incurred during the data processing phase.
  6. Module and Function names are currently limited to 520 characters.
  7. User command-line defined symbol path length should be less than 520 characters and the total symbol path length (environment variable + user command line defined) should not exceed 1040 characters. Exceeding these limits will cause truncation and a warning will be printed.
  8. Synchronizing Kernrate with monitored (user mode) processes with the ‘-q’ command line option will cause Kernrate to immediately stop profiling and process the output when any monitored process exits. This mode incurs less overhead (one additional thread) and has no ill-effects as compared to the debug mode. The –q option allows up to 63 user-mode processes being profiled simultaneously (or less). Kernrate won’t stop the run if this limit is exceeded (but –q won’t have any effect).
  9. Running Kernrate in debug mode will incur higher overhead (see the ‘-nd’, ‘-od’ and ‘-pd’ command line options as well as the ‘-v 16’ and ‘-v64’ verbose output levels below). Note that a separate Kernrate thread will be created for every debugged process for debug event processing. Debug event processing will also slow the debugged process to some extent. This mode allows Kernrate to get debug messages from the system and process them. If one of the monitored processes exits, Kernrate will immediately stop profiling and process the data before all information of the exiting process is lost. The process being debugged will not get killed at the end of the Kernrate run (with the exception of exiting on its own, hitting a fatal exception or if it was created by Kernrate with the ‘-odk’ option, or if running under Win2K OS).
  10. Running Kernrate with the option to suspend the monitored process after profiling is done allows processing the data without allowing the monitored process to exit during that phase. Kernrate attempts to resume the execution of the suspended process before exiting.
  11. Kernrate summaries for average CPU usage are based on system queries that get their data updated via sampling. As explained before, Sampling can miss periodical activities or short term bursts of activity. This could lead to under-estimation of user-time or kernel-time and over-estimation of idle time.  The ‘-v 2’ command line option adds an estimated kernel / user time relative to the total number of hits possible for the time span of the profile (using the profile interrupt sampling as a source instead of the common clock interrupt).  

 

   

Early Exits and Error Messages

 

Kernrate will stop the run in four cases:

  1. Command line error (This will result in printing an explanation of the error and in some cases a brief usage guide).
  2. Memory allocation errors (a proper message will be printed).
  3. Failure in calls to some API’s critical for the success of the run (a proper message will be printed).
  4. Trying to run on an operating system other than NT-based or older than Windows 2000.

 

Kernrate will not produce hits at all (not even for the default source Time) if:

  1. There were no hits.
  2. The particular Hal does not support profile counters or it has a bug.
  3. The process being monitored exits prematurely (unless synchronization or debug mode are used).

 

 

Kernrate may produce no hits on some of the CPU sources specified by the user if:

1.      There were no hits for a particular CPU source.

2.      The specified CPU sources are incompatible to run simultaneously (on i386 and IA64 platforms, but not on AMD64 platform). This case requires switching to cyclic profiling mode.

Notes:

 

1. A typical multi-process profiling command line (including kernel) should look like:

    kernrate -a -z ntoskrnl -z ntdll -z kernel32 -p 1234 -z w3svc -z iisrtl -p 4321 -z comdlg32 –z

    msvcrt ... (other options).

    The first group of -z denotes either kernel modules and/or modules common across processes.

    The other -z groups are process specific and should always follow the appropriate -p xxx (or –n xxx or

    –o xxx).

    There is no need to specify the module extension with the –z option unless there is a

    possibility of ambiguity, such as a .exe and a .dll carrying the same name. Long module names that

    include one or more periods are allowed. Fully qualified path is also allowed.

    Alternatively, one may use the auto-zoom option (‘-si’ command line option) to specify an initial

    profile time that will be used to identify the active modules and zoom on them automatically (see note

    24 below).

2. With the '-n' option, use the common modules -z option if you expect more than one process

    bearing the same name, i.e.    kernrate – z ntdll –z iisrtl –z w3core –n w3wp …..(other options).

    It is assumed that the process has the extension .exe (you can bypass this assumption using the

     –p option).

3. Precedence rules: Only the –z option has precedence rules regarding its location on the command line.

4. No symbol files are needed when doing preliminary profiling (i.e. no zoom modules are

    specified using the –z option). However, correct symbols and symbol path are essential for

    zooming successfully.

5. There is no guaranty that all listed sources (-L or –Lx options) are supported or that they will

    encounter any hits. Kernrate will not fail in that case but will just print a message that no hits

    were encountered for the specific source.

6. The HAL will permit only certain ranges of sampling rates for any given source. Kernrate will

    print the actual sampling rate used for every source and an appropriate note if it failed to set the

    user requested rate. A margin of 25% deviation is currently allowed from the user specified

    rate before attempting to set the rate back to the default one. (Some machines were found to

    exhibit that much of a deviation between the requested rate and the actual set one).

7. The default dbghelp symbol path is (note the dot at the beginning):

    . ;%_NT_SYMBOL_PATH%;%_NT_ALTERNATE_SYMBOL_PATH%;

    Symbol paths can also be specified using the –J option. In that case the specified path will be

    pre-pended to the previous one. ImageHlp may not be able to find the executable file if

    kernrate is being run from some user directory. Kernrate will append

    %WINDIR%\SYSTEM32\DRIVERS; %WINDIR%\SYSTEM32; %WINDIR% to the end of the path

    string. However be aware that the executable found may be the wrong one in case Imagehlp looked into

    some subdirectory (such as a service pack backup one) before looking in the right place. To enforce

    looking for an executable binary in a given directory one may use a fully qualified path with the –z

    option, instead of just the binary name (e.g. –z c:\windows\mydir\somedll.dll instead of –z somedll).

    Do not assume that if the user specified symbol path is %windir%\symbols then ImageHlp will

    automatically look for sub-folders such as \dll, \exe, or \sys. Most of the reported Kernrate zooming

    problems are caused by bad symbol paths. Use the ‘–v’ option to make sure you got the correct

    executables and associated symbol files.

8. Specifying a low minimum lock contention other than the default (1000) for filtering the output could

    cause the system to become unresponsive, and extend the processing time (for the system locks in

    particular). Not all lock addresses resolve to symbols, but use the –v option to make sure that Kernrate

    is looking for these symbols in the right place (see previous note).

9. Use the –k option to reduce the amount of output only after you get familiar with the typical number of

    hits that you get for the sampling rate being used. The –k option does not apply to verbose data. 

10. The '-g' option will attempt to turn on multiple sources (currently 12 in total). One source at a time

     profiling mode will automatically be forced (see remarks regarding the ‘-c’ option), so an adequate run

    time is needed. Output is not guarantied (Hal/driver support dependant). The run will not abort because

    of total lack of, or partial support and hit counts will be printed for supported sources.

11. There is no guaranty that any combination of sources can be turned on simultaneously (on x86

     and IA64 platforms, AMD64 platforms do guarantee that). In addition there is a platform dependent

     limitation on how many sources can be turned on simultaneously (unlimited on AMD platforms, 4 on

     IA64 platforms, 1 on x86 platforms). If the user specified several sources in addition to TIME but did

     not use the ‘-c’ command line option, kernrate will attempt to turn on these sources simultaneously but

     will print a warning message. Kernrate will automatically switch to cyclic profiling of one source, one

     process at a time if the number of sources requested exceeds the maximum number allowed to be

     turned on simultaneously for that platform, using the default switch rate (1000 ms).

12. On IA64 platforms, the user may actually turn off (disable) the default source (ProfileTime) by setting

      ‘-I ProfileTime 0’ or even ‘-I 0’ on the command line. This may be necessary when multiple sources

     are specified, because there is no guaranty any of them will work simultaneously with ProfileTime.

     Setting  ‘-I ProfileTime 0’ or ‘-I 0’ on any platform will always prevent profiling of the default source

     (Time or ProfileTime) even if not actually disabling the source. This can be used to reduce the amount

     of output if CPU Time profile is not required.  

13. Use of the ‘–c’ option (profiling one source, one process at a time) consumes less system resources

     than the default usage (profiling all sources and processes simultaneously).

14. The ‘–w’ option causes kernrate to initialize and wait before starting the actual profile. The ‘–wp’

     option causes kernrate to wait before initialization and only if there is actually any process created by

     kernrate. This wait will be independent from the ‘-oi’ (wait for input idle) option, so if ‘-oi’ is  

     specified, one may opt to use –wp 0 to cancel the other wait. The ‘-wx’ option causes Kernrate to wait

     before exiting. This can be useful with console applications created by Kernrate with the suspend

     mode, that want to exit immediately upon completion. (Kernrate will then pause before resuming the

     main thread of the monitored application). The default wait if –wp is not specified is 2 seconds before

     initializing Kernrate. If  ‘-oi’ is specified, Kernrate will wait up to 10 seconds for input idle for every

     process created by that specific option, immediately after creating the process. Multiple ‘-o’ entries are

     allowed, but only those that specify the ‘i’ sub-option will wait for input idle. Console processes are

     not affected by the ‘-oi’ option.

15. Allowed redirection formats within the curly brackets of the ‘-o’ command line option are:

      (a) Any of:  ^<InputSream 1^>OutputStream  2^>ErrorStream (output and error streams allow

            appending as well, e.g. 1^>^>OutputStream).

      (b) Any of  ^<InputSream ^>OutputStream (output appending allowed as well).

      Also:

  • Piping (‘|’) is not supported as an option within the curly brackets.
  • When redirection is used, the newly created process window may appear empty (blank).
  • Redirection of any output stream within the curly brackets is not allowed if the user specified

                  more than one process to be created, by using the ‘-o Number ProcessName {parameters}’

                        command line option.

16. Stopping the Kernrate run can be done automatically by specifying the ‘-s Seconds’ command line

      option, or it can be done manually by pressing ‘ctrl-c’. In some situations one may be unable to use

      ‘ctrl-c’ because the control handler is taken by another process. In such cases one has to use the ‘-s

      Seconds’ option.

17. To construct a processor affinity mask for the ‘-m’ option select the indices of the processors to

      profile, do 1 << ProcessorIndex for each processor, convert to hex and add together (the index of the

      first processor is 0). There is no point in using the mask if the processes are not pre-bounded to

      processors. By default, all processors are profiled when ‘-m’ is specified without a mask. Bogus

      processor entries in the mask will be ignored.

18. The '-nd' '-ns' '-od' '-os' options also allow specifying 'v' and '#' or any other valid sub-options, e.g.

      '-nsv# N ProcessName’.

19. The '-pd' '-ps' options also allow specifying the 'v' sub-option, e.g. '-psv ProcessId'.

20. If debugging mode is selected ('-nd', '-od' or '-pd') the debuggee will not get killed upon Kernrate exit

      (unless it exits on its own or hits a fatal exception, or if running under Win2K OS). The ‘-odk’

      option forces processes created by Kernrate to get killed upon Kernrate exit (applying only to the

      process or processes created under that specific instance of ‘-o’).  

21. If suspend mode is selected ('-ns', '-os' or '-ps') Kernrate will attempt to resume the suspended process

      after post processing is done.

22. Kernrate cannot prevent a GUI or a console from getting destroyed upon exit of their corresponding

      process. Kernrate merely tries to keep the exiting process address space long enough to process the

      profiling data.

23. Some process-specific information (see output notes below) may be unavailable for a process that

      exits early, even with synchronize (‘-q’) or debug (‘-nd’, ‘-od’ or ‘-pd’) modes.

 

24. Auto-Zoom Profiling Mode:

      This option saves the need for an initial run to identify the most active modules. It also saves on   

      command line input. This option is selected by specifying an initial-profile time ('-si seconds'), used

      to identify the active modules in a process. The overhead of using this option is very small. It is not

      recommended for use with short-lived processes.

      Notes:

  • This is a global option (a user may force adding a module to the zoom list by specifying

'-z module_name' entries).

  • Delay options ('-w') will be executed before the initial profile in this mode.
  • Inter-process synchronization: KERNRATE_READY event will be set before starting the initial

profile in this mode.

  • The ‘-km MinPercentHits' option can be used to filter-out zooming on modules that scored less than MinPercentHits in the initial profile.
  • A kernel profile in this mode will likely pick-up and zoom on the idle-process (processor driver)

module as well.

      Usage Example:

      A 10 second initial-profile, followed by a 60 seconds profile with zoom of both the Kernel and a

       User-Mode process called myproc: “kernrate -a -n myproc -si 10 -s 60”.

 

25. Wild-Card Zoom-All:

      This option saves the need for an initial run to identify the most active modules. It also saves on

      command line input. The overhead of using this option can be very high in terms of memory footprint,

      initialization time and CPU usage. If possible, avoid this option altogether and select the Auto-Zoom

      Profiling Mode instead.

      Notes:

      This is a global option (it ignores the Common and Per-Process Zoom Lists rule).

      Usage Example:

      A 60 seconds profile with zoom of both the Kernel and a User-Mode process called myproc:

      “kernrate -a -z * -n myproc -s 60”.

Managed Code .NET Support

 

Kernrate can profile managed code JIT (Just In Time compilation) modules as well as pre-compiled (NGEN) modules. The support requires installing a helper library (IP2MD.DLL, not a part of Kernrate).

Kernrate will search for the helper library (and any library that ip2md.dll may depend on) in the current directory, then in the directory where the currently running Kernrate executable is located, then in system32 and finally in syswow64.

No special command line switches are required to activate that support. Kernrate will assign unique names to each JIT module (because otherwise there may be more than one module bearing the same name). To zoom into any JIT module, the user needs to specify its unique name after the ‘–z’ option (multiple ‘-z’ usage is allowed). Kernrate also allows to use a generic option for zooming on all JIT modules in a process, by specifying ‘–z jit*’ instead of multiple specific ‘–z’ entries (i.e –z jit0 –z jit1 –z jit2 …). Pre-compiled (NGEN) modules should be treated as any regular module. At the end of the output for each process Kernrate will print a summary of the JIT modules status (Exists, New or Gone) based on snapshots taken before and after the profile. If a module is marked “GONE” do not fully trust the data related to it. Kernrate may miss some short-lived JIT modules altogether if they were created after the profile started and were destroyed before the profile ended. Managed code modules can be quite large and take a long time to process for symbols. One may increase the bucket size to 32 or 64 bytes using the ‘-b 32’ or ‘-b 64’ command line option to reduce processing time (default is 16 bytes). Increasing the bucket size on the other hand may cause symbol misses (minimum bucket size is 4 bytes). Another option is to use a faster enumeration (‘-fs’) instead of increasing the bucket size. The faster enumeration may miss some symbols or bucket-sharing details. In some rare cases Kernrate may seemingly “hang” while processing a managed code module if the module is gone while being processed. In such a case it is suggested to “ctrl-c” the kernrate run and either use whatever data has been gathered so far or try re-running. 

 

Always install updated debugger tools before running Kernrate to profile managed code (In particular, dbgeng.dll and dbghelp.dll). On Windows 2000 those need to be replaced with the XP version or later. To replace those modules one needs to disable system file protection (SFP), or install them with a tool that handles SFP), or install them in the directory where Kernrate is located.

 

Profiling 32 Bit Applications Running on a 64 Bit System (WOW64)

 

The 64 Bit versions of Kernrate support profiling of 32 Bit applications running on 64 Bit Systems (Wow64). The support is transparent and does not require any input from the user. The output will tag 32 bit modules clearly, to distinguish them from native 64 Bit modules. If the user zoomed in on a module that has both native and 32 Bit versions, using just the module name, then both versions will get zoomed (e.g. –z ntdll). To zoom only on a particular version of the module the user needs to specify the –z option with a fully qualified path (e.g. –z c:\windows\syswow64\ntdll.dll).

Notes and Limitations:

  1. On X64 systems, the 64-bit Kernrate cannot zoom on 32 bit managed code modules (running on WOW64).
  2. On X64 systems, the 32 bit Kernrate will run on WOW64 and zoom on 32 bit managed or non-managed code modules, however it cannot be used to profile the 64 bit kernel. Note that in this case all X64 CPU events will be supported.

 

Synchronizing the Profile with another User Application

 

Kernrate profiling can be triggered and also stopped from another user application, using a synchronization mechanism of named events. This is useful for automation and profiling selected pieces of code. The default named events are:

 

  1. “Global\\KernrateReady”         - Set by Kernrate when it is done initializing and ready to profile.
  2. “Global\\KernrateStart”                        - Set by the application to trigger Kernrate to start profiling.
  3. “Global\\KernrateStop”                        - Set by the application to indicate to Kernrate to stop profiling.
  4. “Global\\KernrateProcessData” -  Set by the application to indicate to Kernrate to proceed with  

                                                             data processing and output.

  1. “Global\\KernrateDone”          - Set by Kernrate to signal the application that it is done

  processing, so the application can exit safely if it wishes.

 

The triggering application needs to:

  1. Create four named events using the names in the list above (with fManualReset set to TRUE).
  2. Wait on the “Global\\KernrateReady” event handle to get signaled.
  3. Set the “Global\\KernrateStart” event when it wants Kernrate to start the profile.
  4. Set the “Global\\KernrateStop” event when it wants Kernrate to stop the profile.
  5. Set the “Global\\KernrateProcessData” event when it finished its own performance-critical part of the run.
  6. Wait on the “Global\\KernrateDone” event handle to get signaled before exiting.

 

It is possible to specify the event names instead of using the generic ones, by utilizing the ‘-en Event Name’ command line option (where n=1-5). In that case each of the ‘–en’ options can be specified only once followed by an event name. All five events must be specified. The event names should be distinct from one another and rather unique for the system (no need to specify Global\\ before the event name).

Example: -e1 KrReady –e2 KrStart –e3 KrStop –e4 KrProcessData –e5 KrDone. The application synchronizing with Kernrate must open the appropriate events pre-pending “Global\\” to the name of each event.

 

Notes:

  1. At this point the cycle above cannot be repeated using the same Kernrate process.
  2. The user must add the ‘-w’ option to the Kernrate command line (no optional wait period) for the synchronization to work.
  3. If profiling the system process alone, there is no importance which process gets started first (Kernrate or the triggering application).
  4. If profiling a user-mode application, one must start the application being profiled before Kernrate, or launch it via Kernrate itself (see the ‘-o’ command line option). The triggering application can then be either the profiled application itself or a separate application.
  5. Kernrate will also accept a profile period definition (see the ‘-s’ command line option). In that case, it will stop the profile either when the period expires or when the “Global\\KernrateStop” event is signaled, whichever comes first.
  6. Failure to set any of the events that the application is supposed to set may cause Kernrate to hang waiting.

 

 

 

Added Measures for Increased Confidence in the Results

 

Kernrate can neither profile modules loaded after profiling has started, nor pieces of code in the process address space that are not pre-identified as a module or a managed code range. To increase the confidence in the results, Kernrate creates a fictitious, one-bucket module that covers all or a significant part (OS limitation) of the process address space. Hits that occur in that range that do not belong to any known module count towards the “UNKNOWN_MODULE(s). A significant number of hits in that special module indicate either libraries loaded after profiling has begun or some unaccounted for managed code or code range.

 

The address range covered by the special module is printed out for each process. The special module applies only to user-mode processes and although it appears in the process module list, it cannot be zoomed into.

 

Kernrate will issue a warning if it suspects bad symbols or if it could not find any symbols (use the ‘-v 1’ verbose mode to get more details in such cases).

 

The user can force Kernrate to exit in case bad or no symbols are detected or a failure to initialize support for managed code (using the ‘–fx’ option).

Address Instruction Operands (tab delimited so that the operands can contain commas). Entries that are associated with he bucket itself will be marked with an arrow (à) to the left of the line.

Raw-Data Output File

 

The ‘-yr filename’ command line option will create a raw data, tab delimited output file for later processing by a spreadsheet program like excel for example. The raw data file will be created in the same directory from where Kernrate is being run. If no file name is specified, the name of the file will be of the form KRxxx.txt where xxx is a number limited to 200 entries. After 200 files of this type are created, the next run of Kernrate will issue an error message asking the user to clean up (no raw data output will be generated). If the user deleted a raw data file from the sequence then the next run will pick up and create a file replacing the first missing entry and so on. If the user specified a file name, Kernrate will attempt to create this file in the current directory (overwriting any previous file by that name).

 

The raw data output file consists of header lines and of data lines. The header line starts with the word HEADER: followed by a name identifier and then by a description of the contents of the data lines corresponding to that header. A header line can appear more than once (serving also as a separator between chunks of data). A header line can appear without any corresponding data lines.

 

The following header lines entries currently exist:

 

HEADER: MachineInfo, Element, Value

HEADER: TaskList, ProcessID, ProcessName, Status, %KernelTime, %UserTime, HostedServices

HEADER: TraceInfo, Element, Value

HEADER: TraceTotals, Element, Value, Units,

HEADER: KernrateOptions, Option, SubOptions, Value1, Value2, Value3

HEADER: MonitoredProcesses, ProcessName, PID

HEADER: ZoomList, Type, PID, ModuleName

HEADER: ProcessorSummary, Processor, Elapsed, Kernel, User, Idle, DPC, Interrupt, TotalInterrupts

HEADER: SystemSummary, Element, StartValue, StopValue

HEADER: UModeProcessSummary, ProcessName, PID, Element, Value

HEADER: UModeProcessDetail, ProcessName, PID, Element, StartCount, StopCount

HEADER: ModuleSummary, Type, ModuleName, Pid, ProfileSource, ProfileTime, CPU, ModuleHits, Wow64

HEADER: ModuleDetail, Type, ModuleName, Function, Pid, ProfileSource, ProfileTime, CPU, CertainHits, UpperHits, Wow64

HEADER: ModuleRawData,  Pid, ProfileSource, Module, Function, CPU, Bucket, Displacement, Address, Hits, SourceLine, SourceFile, FirstInstAddr, Type, Wow64

HEADER: JITModuleRawData, Pid, ProfileSource, Module, Function, CPU, Bucket, Displacement, Address,Hits, SourceLine, SourceFile, FirstInstAddr, Type, Wow64

HEADER: BucketShareInfo, Pid, ProfileSource, Module, Function, CPU, Bucket, Displacement, Address, Hits, SourceLine, SourceFile, FirstInstAddr, Type, Wow64

HEADER: SystemLocks, Pid, Type, Address, ContentionCount, Status, ModuleBase, Module, Symbol

HEADER: ProcessLocks, Pid, Type, Address, ContentionCount, Status, ModuleBase, Module, Symbol

HEADER: ProfileSourceInfo, Name, Rate, DefaultRate, Unit

HEADER: WARNING, MessageBody, Detail1, Detail2

 

Data lines start with the word Data: followed by the appropriate header name and the corresponding data.

Data entries may contain numbers or strings. For example, some data entries may appear as NA for “not available” or “not applicable” (this may replace number entries as well). Some other conventions used are K for kernel, U for system, T for total, JIT for jitted managed code modules, PREJIT for pre-jitted modules. Hex values are preceded by “0x”. Some of the number entries are 64 bit long. Note that at present, data for the headers “ModuleRawData”, “JITModuleRawData” and “BucketShareInfo” will be generated only with the ‘-r’ command line option (and not with the –v 0x20 option).

 

In processing the raw data it is important to use the correct normalizing quantities when calculating percentages. In particular, note that the total elapsed profile time for a process or a module may be different than that of the whole run if cyclic profiling is turned on either automatically or by the user (see the ‘-c’ command line option). The number of total possible hits for calculating hit percentages should be scaled accordingly from the total possible hits for the run.

 

Another important processing factor should be the correct choice of the context. For example, data for a zoomed module may appear several times in the context of different processes or in the context of different profile sources within the same process. The field “Wow64” is used to tag 32 Bit modules running on a 64 Bit system (WOW64) and to distinguish them from native 64 Bit modules that may carry the same name.

f.       The –v 2 option will also print the full list of modules loaded for each process and a final list of processes running at the end of the profile.

Command line:

 

Kernrate –a –m -si 10 –s 60 –n myproc    

 

Notes:

a.     Adding ‘-km 1’ to the command line above will limit zooming to modules that score at least 1% of  

       the total hits in the initial profile.

Kernrate –lx

 a.  The short names from this list are used for specifying the profile sources in the following case.

Get Imagehlp verbose symbol information. Get summary of %CPU usage for all processes running on the system during the profile, get lock contention information for the system.

 

Command line:

 

Kernrate –v –f  -t -xk –b 4 –w –s 120 –m –c –r –z ntoskrnl –z hal –z tcpip –z ndis –I ProfileTime 2000

–I MercedInstRetired 100 –I MercedBranchInstructions 1000 –I MercedBranchStallCycles

 

Notes:

a.       See also the –d and –v 2 options as alternative presentation to the –r option.

b.      The –f option will alow Kernrate to continue and run at high priority during data processing. This can be useful on busy systems when momentary overhead is not an issue.

c.       “ProfileTime” is the default profile (on some other systems it is called “Time”). The sampling rate for this source can also be specified without specifying the source name (i.e. –I 2000).

d.      The last source (MercedBranchStallCycles) has no rate specified with it, meaning that the user chose the default rate.

e.       A rate of 0 (zero) will disable the source (except for the default Time source that is always enabled).

f.       The selection of the –c option in this case will cause the 120 seconds of profile time to be equally divided between the four sources, so each will be sampled for an overall of 30 seconds (the sampling will rotate continuously between the sources, changing sources every 1000 ms unless the user selected a different rotation period with the –c option).
