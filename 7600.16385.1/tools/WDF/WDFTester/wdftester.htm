WdfTester: WDF Driver Testing Toolset

(preliminary version)

 

 

SUMMARY

 

WdfTester is a set of tools that allows testing, verifying and helping debug WDF drivers. The toolset provides a WMI programming interface that could be used in a script or a compiled application. Currently it consists of following two tools:

 

1)     WDF DDI Fault Injection Tester (WdfFiTester)

2)     WDF Driver Call Tracer (WdfCallTracer)

 

WdfFiTester can be used for injecting KMDF DDI failures in KMDF drivers and test the capability of the driver in handling these failures gracefully. This tool can be configured to fail any KMDF DDI that returns NTSTATUS code. There are 175 DDIs in KMDF version 1.7 that return NTSTATUS code.

 

WdfCallTracer can be used for tracing and viewing driver communication with framework in real-time.

 

This document provides information on how to install and use the toolset, and contains a reference section for WMI interface.

 

 

WDF DDI Fault Injection Tester (WdfFiTester)

 

A typical KMDF DDI call has the following pattern:

 

    //
    // Create the device object.
    //
    status = WdfDeviceCreate(
                             &DeviceInit,
                             &attributes,
                             &device
                             );
    if (!NT_SUCCESS(status)) {
        return status;
    }

The DDI returns NTSTATUS code and the driver checks for the return code before proceeding ahead. Many bugs occur because of missing or incorrect check for return code. These bugs may result in unexpected behavior or a bugcheck. For example, a bugcheck may occur if the DDI has an OUT pointer parameter that happens to be NULL and is touched by the driver without checking for the status correctly. This tool allows failing the DDI that has been configured for fault injection by returning NTSTATUS code of STATUS_UNSUCCESSFUL. The driver is expected to handle the failure gracefully. The tool provides a WMI interface for configuring DDI fault injection for a specific driver. As such this WMI interface can be used in a script (vbscript or jscript) or any other user mode app (C, C++, C# and so forth) that can make calls to WMI.

 

Apart from other operations, the tool’s WMI interface allows getting a list of DDIs that were called by a specific KMDF driver, and waiting on a WMI event that gets fired every time a DDI fault injection completes successfully.

 

Installation

 

Follow the steps below to install the tool.

 

1)     Copy following files from WDK (tools\wdf\wdftester\<platform>) to a local folder

 

Wdftester.sys

Wdftester.inf

Wdftester.ctl

Wdftester.tmf

WdftesterScript.wsf

 

2)     Open a command window (with admin privileges on Vista) and execute the following command

          

C:\>cscript WdfTesterScript.wsf install

 

      This will install wdftester driver and start the service.

 

 

Enabling Fault Injection

 

The tool provides a WMI interface for configuring DDI fault injection for a specific driver. This WMI interface can be used in a script (vbscript or jscript) or any user mode app that can make calls to WMI. The tool uses following methods exposed by WMI class named WdfTester to enable and configure DDI fault injection.

 

RegisterDriver (DriverName)

 

ConfigureDdiFi (DdiIndex, Enable, FailureType, DriverName)

 

UnregisterDriver (DriverName)

 

 

A typical fault injection test would consist of the following steps (as coded in a script):

 

1)     Disable the driver in device manager (or unplug the device)

2)     Register the driver with WdfTester by calling the WMI method RegisterDriver( )

3)     Enable fault injection for a specific DDI that is called in your driver by calling WMI method ConfigureDdiFi( )

4)     Start asynchronous WMI event notification

5)     Execute any action that causes execution of the DDI configured for fault injection (e.g. plug-in a child device, enable the driver, disable the driver, unplug the driver, power down device etc.)

6)     Wait for the WMI Event for the DDI to occur

7)     Ensure the failure was handled gracefully.

 

There is script named WdftesterScript.wsf that could be used from command-line to register/configure/unregister driver. Also it has a command line option called “runtest” that does the following. Note that for this option to work you must have devcon.exe present in the same folder as other wdftester files.

 

1)     Register Driver with wdftester (assuming the driver is installed, if not you must first install the driver)

2)     Enable Driver Verifier for this driver (on Vista it does not require reboot)

3)     Disable the driver using devcon.exe

4)     Enable the driver using devcon.exe

5)     Get list of DDIs that were called during the above operation (disable/enable) and that could be failed (only those that return NTSTATUS could be failed). This uses GetDdiCallInfo() method of WdfTester WMI class.

6)     Start asynchronous WMI Event notification

7)     For each DDI that could be failed (from above list)

a.     Configure the DDI for failure (using ConfigureDdiFi() method

b.    Do disable/enable of driver using devcon. This will call the DDI and wdftester will fail the DDI.

c.     Wait on the WMI event to be fired.

d.    If the WMI event is fired, goto step 7 and repeat for next DDI in the list    

8)     Unregister driver

 

 

WDF Driver Call Tracer (WdfCallTracer)

 

Using this tool you can view the DDI and Event calls in real-time. Below is a snapshot of the driver communication for Static Bus Driver Sample (statbus.sys available in WDK). Currently only the DDI calls can be viewed.

 

Follow the steps below on your test machine to view driver communication with KMDF:

 

1)     Install WdfTester using steps described in earlier section.

1)     Launch TraceView.exe from the WDK (<WDK root>\tools\tracing\i386)

 

a.     Select FileàCreate New Log Session

b.    Click AddProvider

c.     Select CTL (Control GUID) File and using file browser select wdftester.ctl file from the folder containing WdfTester files.

d.    Click OK in Provide Control GUID Setup dialog box

e.     Select Select TMF Files and press OK.

f.     Press Add, browse to the folder containing wdftester files and select wdftester.tmf. Press Done when you see wdftester.tmf added to the display window.

g.    Click Next in the Create New Log Session dialog box.

h.     Click Finish in the Log Sessions Options dialog box.

 

Now you are ready to view the driver communication.

 

2)     Register your KMDF driver from command-line using WdftesterScript.wsf script.

 

             cscript WdftesterScript.wsf register statbus.sys

 

3)     Now enable the driver from device manager or plug-in your hardware. If your driver was already enabled then disable it and enable back from device manager.

 

You should now see driver communication in the TraceView app.

 

 

 

WMI Interface Reference

 

 

WdfTester Class

 

An instance of the WdfTester WMI class is created by the kernel module—a component of WdfTester tool set. It exposes interfaces that can be used to configure DDI fault injection. You will need to install the tool (including starting the service) before you can see an instance of this class.

 

class WdfTester : WdfTesterBase {

    uint32 RegisteredDriversCount;

    string RegisteredDriversList[];

};

 

Methods

The WdfTester class defines the following methods.

 

Method

Description

RegisterDriver

Enable testing for the driver. This is required for configuring fault injection.

UnregisterDriver

Disable testing for the driver.

ConfigureDdiFi

Configure fault injection for a DDI

  

 

RegisterDriver Method

The RegisterDriver method enables the KMDF driver for testing which means that the underlying kernel module takes appropriate action to enable any future test (such as fault injection) for the driver. This method also enables the ability to visually monitor the communication of client driver with the framework using a WPP trace log viewer such as TraceView.exe.

 

void RegisterDriver( string DriverName,

                     uint32 Status

                     );

Parameters

DriverName

[in] Name of the driver image with extension (e.g. statbus.sys).

Status

[out] Return value as returned by the underlying kernel module.

Return Value

Void since the return value for the method is returned in the Status parameter.

 

 

UnregisterDriver Method

The UnregisterDriver method disables the KMDF driver for testing which means that the underlying kernel module takes appropriate action to remove the driver from its list of drivers to be tested. This method also disables the ability to visually monitor the communication of client driver with the framework using a WPP trace log viewer such as TraceView.exe.

 

void UnregisterDriver( string DriverName,

                       uint32 Status

                       );

Parameters

DriverName

[in] Name of the driver image with extension (e.g. statbus.sys).

Status

[out] Return value as returned by the underlying kernel module.

Return Value

Void since the return value for the method is returned in the Status parameter.

 

 

ConfigureDdiFi Method

The ConfigureDdiFi method configures fault injection for the given DDI and the driver. This method is used to enable or disable fault injection on a specific DDI. The configuration becomes effective immediately after a successful call (no need for restarting the driver).

 

void ConfigureDdiFi( uint16 DdiIndex,

                     Boolean Enable,

                     uint16 FailureType, 

                     uint32 Status

                     );

Parameters

DdiIndex

[in] Index of the DDI (from WdfFuncEnum.h in <WDK root>\inc\wdf\kmdf\10 folder).

 

Enable

[in] TRUE if enabling DDI failure, FALSE if disabling DDI failure.

 

 

FailureType

[in] Type of failure (currently unused).

 

Status

[out] Return value as returned by the underlying kernel module.

Return Value

Void since the return value for the method is returned in the Status parameter.

 

GetDdiCallInfo Method

The ConfigureDdiFi method configures fault injection for the given DDI and the driver. This method is used to enable or disable fault injection on a specific DDI. The configuration becomes effective immediately after a successful call (no need for restarting the driver).

 

void GetDdiCallInfo ( string Name,

                     uint32 ArraySize,

                     DdiCallInfo Array[387], 

                     uint32 Status

                     );

Parameters

Name

[in] Name of driver binary (e.g. statbus.sys)

 

ArraySize

[out] Size of array DdiCallInfoArray

 

 

Array

[out] array of DdiCallInfo class. Array index represents the DDI index.

 

Status

[out] Return value as returned by the underlying kernel module.

Return Value

Void since the return value for the method is returned in the Status parameter.

 

 

DdiCallInfo Class

 

This class is used by GetDdiCallInfo method of WdfTester class.

 

class DdiCallInfo

{

     boolean Called;

     uint32 CallCount;

};

 

 

WdfTester_DdiFiEvent Class

 

This event is fired when a DDI fault injection occurs. The event carries information about the DDI and the Driver where fault injection occurred.

 

class WdfTester_DdiFiEvent : WdfTesterEvent

{

     uint32 DdiIndex;

     string DriverName;

};
